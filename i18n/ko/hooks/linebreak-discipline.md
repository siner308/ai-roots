# Linebreak Discipline Hook

새로 추가된 Markdown 줄이 문장 중간에서 hard-break 되는 것을 잡아내는 `Edit|Write|MultiEdit` 대상 `PostToolUse` hook.
강제하는 규칙: 린터나 포매터가 폭에 에러를 내지 않는 한, hard break는 문장이 끝나는 자리에만 올 수 있다.
문장마다 줄을 바꿔야 하는 것은 아니다 — 한 줄에 여러 문장이 있어도 된다.

## 왜 존재하나

prose-style 규칙은 이미 문장 중간 hard-break를 금지하지만, 실전에서는 더 강한 관성에 졌다: 이미 특정 컬럼으로 hard-wrap 된 파일을 편집할 때 기존 스타일을 따라 하는 것이 좋은 매너처럼 읽혀서, 새 텍스트가 컬럼 한계에서 감긴다 — 관사가 명사와 떨어지고 괄호 묶음이 쪼개진다.
resident 규칙은 컨텍스트 전체와 경쟁하는데, 주변 파일이 잘못된 행동의 본보기가 되는 바로 그 순간에 진다.
`comment-discipline`을 hook으로 만든 것과 같은 실패 모드다.

## 무엇을 하나

모든 Markdown 편집에서 추가된 텍스트만 검사한다 (Write의 content, Edit/MultiEdit의 새 문자열 — 파일의 기존 감김은 절대 발동시키지 않는다).
줄이 문장 종결 부호 없이 끝나고 다음 줄이 같은 문장을 이어가면 플래그된다.
결과는 `decision: "block"`으로 돌아와, 모든 hard break가 문장 경계에 오도록 줄을 다시 합칠 것을 요구한다.

명시된 예외: 프로젝트에서 줄 폭에 에러를 내는 린터나 포매터 — 도구 이름을 대고 감김을 유지한다.
파일의 기존 감김 스타일이나 현재 디스플레이 폭은 예외 사유가 아니다.

## 무엇을 건너뛰나

- Markdown이 아닌 파일, fenced code block, YAML frontmatter.
- 구조 줄: 제목, 표 행, fence 마커, `---` 구분선.
- 다음 줄이 새 블록(제목, 리스트 항목, 표, 인용, fence)을 시작하는 줄 — 그 끊김은 감김이 아니라 실제 경계다.

## 알려진 한계 (검토 후 수용)

파서가 아니라 휴리스틱이다: 종결 부호 없는 약어로 정당하게 끝나는 문장이나, 절 경계에서 의도적으로 끊는 semantic line break 스타일도 플래그된다 — verdict 방식의 block이라 사유를 대고 유지할 수 있다.
Edit 조각은 내부만 검사하므로, 조각의 마지막 줄이 파일의 기존 텍스트와 문장 중간에서 이어지는 경우는 놓친다.
지배적 실패(새 문단 전체를 감아 쓰는 것)는 커버된다.
