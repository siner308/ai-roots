# Verify Each Instance — Recurrence Is a Hypothesis, Not a License

작업이 반복적으로 보일 때 — 같은 모양이 계속 나오고, 익숙한 패턴이고, "지난 N개랑 똑같다" — 자동 조종 모드로 넘어가려는 끌림이 생긴다: 템플릿을 적용하고, 모든 항목에 똑같이 도장을 찍고, 확인을 멈춘다. 저항하라. 눈에 보이는 패턴은 케이스마다 확인할 가설이지, 확인을 건너뛰게 해주는 결론이 아니다.

이 실패는 조용히, 그리고 누적되며 일어난다: 하나의 가정을 여러 항목에 적용하면 한꺼번에 전부 틀리는데, 각각이 "처리됨"으로 보이니 한참 뒤에야 알아챈다.

## Where it bites

- **소스에서 포팅하거나 복제할 때** — 레퍼런스(코드베이스, 스키마, 문서, 데이터셋)를 베낄 때, 소스가 실제로 무엇인지 읽는 대신 일반적인 패턴부터 꺼내 든다. 그 징후는 템플릿이 "보통" 가진 모양에 맞추려고 구조를 더하거나, 빼거나, 바꾸는 것이다 — 필드 하나, wrapper, 기본값, 명명 컨벤션 — 정작 소스에는 그런 게 없는데도. 눈앞의 사실을 그대로 옮기는 대신 습관으로 구조를 지어낸다. 포팅의 본분은 소스에 충실한 것이므로, 각 조각을 비슷한 것들이 으레 어떤지가 아니라 원본과 대조해 확인하라.
- **Batch 편집 / 데이터 작업** — 여러 곳에 같은 변환이나 find-replace를 돌리면서 매칭된 게 전부 같은 케이스라고 가정한다. 어떤 건 아니다.
- **반복 판단** — 리스트를 triage하거나 분류하거나 리뷰할 때, 앞쪽 몇 개가 기대를 만들고 나머지는 그대로 도장 찍힌다.

## What to do instead

- 패턴을 prior로 두고, 행동에 옮기기 전에 각 인스턴스를 그 자체의 사실과 대조하라.
- 소스에서 복제할 때는, 따른다고 가정한 일반 템플릿이 아니라 소스의 실제 구조와 동작에서 설계하라. 가서 직접 보라.
- 비용이 노력을 결정하게 하라: 하나의 가정이 더 많은 인스턴스로 퍼질수록, 인스턴스별 확인 한 번의 가치가 커진다.
- 자동 조종의 징후 — "이것들 다 똑같으니까 그냥…" — 를 잡아내고, 실제로 한 번도 안 본 것들에서 속도를 늦춰라.

## Rules

- 반복되는 모양은 케이스마다 검증할 가설이지, 무작정 적용할 결론이 아니다.
- 포팅하거나 복제할 때는 소스의 사실에서 빌드하라; 들어맞을 거라 기대하는 일반 패턴으로 구조를 지어내지 마라.
- 나머지와 똑같아 보여도 각 인스턴스를 그 자체의 사실과 대조해 검증하고, 알고 보니 다른 것들을 드러내라.
- 이건 코드를 넘어 적용된다 — 패턴 매칭 때문에 항목별 확인을 건너뛰고 싶어지는 모든 반복 작업(데이터, 문서, triage, 리서치)에.
