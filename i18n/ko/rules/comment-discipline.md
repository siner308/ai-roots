# Comment Discipline

기본값: 주석을 쓰지 않는다. 좋은 이름, 작은 함수, 명확한 흐름이 이미 코드가 WHAT을 하는지 말해준다. 주석은 코드가 표현할 수 없는 것 — 비자명한 WHY — 만 더한다.

주석과 docstring은 필수가 아니며, 주석이 없는 함수가 미완성인 것도 아니다. 주석이 필요 없는 깔끔한 코드가 목표이지, 채워 넣어야 할 빈칸이 아니다. 주석이나 docstring을 단지 "없으니까" 붙이지 마라 — 없는 게 올바른 기본값이고, 있는 쪽이 스스로 정당화해야 하는 예외다.

이건 어디에나 적용된다: 메인 세션 편집, 서브에이전트가 만든 코드, 리팩터링, 버그 수정. 구현을 위임할 때는 브리핑에 이 규칙을 다시 적어라 — 약한 모델은 제약을 반복하지 않으면 방어적 주석 습관으로 회귀한다.

## 쓸 가치가 있는 주석은 이것뿐

다음 중 하나일 때만 주석을 써라. 이 목록은 닫혀 있다: 쓰려는 것이 여기에 명확히 해당하지 않으면 지워라.

- **숨은 제약 / 선행조건** — 이 코드가 가정하지만 보여줄 수 없는 불변 조건: `// caller holds the lock`, `// must run inside a transaction`.
- **Workaround** — 이 이상한 코드가 왜 존재하는지, 업스트림 수정이 들어오면 떼어낼 수 있도록 링크나 이슈를 붙여서.
- **틀려 보이지만 맞는 코드** — 이유를 알기 전엔 틀린 것처럼 보이는 코드. 그 이유를 적어라.
- **미묘한 불변 조건** — 읽는 사람이 무심코 깨뜨릴 수 있는 순서, idempotency, 수치 정밀도 가정.

자가 점검: "이걸 지우면 주의 깊은 독자가 *헷갈리거나 놀랄까*?" 아니라면 지워라. 기준은 혼란이지, 블록이 휑해 보이는 것이 아니다.

## 그 외 전부: 주석 없음

금지 주석 유형 목록을 들고 다니지 마라 — 그 목록은 끝이 없고, 다음 나쁜 주석은 항상 항목들 사이로 빠져나간다. 규칙은 하나다: **위 허용 목록에 없으면 → 쓰지 마라.**

코드가 과하게 주석 달리는 익숙한 방식들은 외워야 할 별개 규칙이 아니다 — 전부 같은 실패(목록에 없음)다: 다음 줄이 WHAT을 하는지 다시 적기, 시그니처 echo, 작업이나 PR 서술, *호출자나 다른 레이어*가 결과로 뭘 하는지 설명, 스크래치패드 메모, 지운 코드의 묘비, 섹션 구분선. 열거하지 말고 그냥 목록을 적용해라.

방어적 주석으로 흘러가는 신호: 블록이 "휑해 보여서", 주석이 다음 세 줄을 풀어 쓴 것, 가상의 리뷰어에게 서술하는 것, 또는 모든 함수에 무조건 docstring을 다는 것. 알아채면 지우고, 코드를 믿어라.

## Rules

- 주석 없음이 기본값이고, 주석이나 docstring은 절대 필수가 아니다. 주석 없는 깔끔한 코드가 완성된 상태이지 미완성이 아니다 — 없다는 이유만으로 붙이지 마라.
- 주석은 허용 목록의 정당화가 필요하다; 침묵에는 정당화가 필요 없다.
- 주석이 정당화되면 한 문장으로 WHY부터 적어라.
- 주석은 THIS 단위 자신의 비자명한 사실을 문서화한다 — 호출자나 다른 레이어가 뭘 하는지의 지도가 아니다 (그건 PR 본문이나 커밋 메시지에 들어갈 내용이다). 단위가 가정하는 선행조건은 그 단위 자신의 계약의 일부이므로 남는다.
- 구현을 서브에이전트에 브리핑할 때 이 규칙을 다시 적어라 — 위임을 거치면 안정적으로 보존되지 않는다.
- docstring도 같은 규율. 시그니처를 풀어 쓰는 건 소음이다. 예외: 언어 도구가 공개 API 문서를 강제할 때(Go `revive`, Rust `missing_docs`, Python `pydocstyle`) 노출된 식별자에 한 줄 계약 설명은 괜찮다 — 시그니처가 아니라 계약을 적어라.
