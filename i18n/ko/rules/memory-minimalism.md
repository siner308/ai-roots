# Memory Minimalism

자동 memory 시스템은 항목을 기기 한 대의 파일시스템에 저장한다. 머신, 프로젝트, 작업 맥락 사이에서 동기화되지 않는다. 같은 사용자가 여러 기기에서 일하거나, 본 프로젝트와 사이드 프로젝트를 오가거나, 공유 rule 저장소를 서로 다른 환경에서 개선하면, memory 항목은 조용히 갈라진다 — 한 세션에서 저장한 게 다음 세션에서는 보이지 않고, 단일 세션 안에서는 이 비대칭을 알아챌 수 없다.

내용이 재사용 가능한 거라면, memory 항목보다 버전 관리되는 견고한 산출물(rule 파일, `CLAUDE.md`, 프로젝트 문서)을 우선하라.

## memory를 쓰지 말아야 할 때

- **컨벤션, 원칙, 행동 규칙** — 이런 건 `rules/`(범용)나 프로젝트 `CLAUDE.md`(프로젝트별)에 속한다. 둘 다 버전 관리되고 모든 기기에서 동일하다.
- **대화보다 오래 사는 프로젝트 사실** — 프로젝트 문서, `CLAUDE.md`, 또는 전용 rule 파일에 속한다.
- **공유 저장소를 위한 rule 작성** — rule을 어떻게 쓰는지에 대한 메타 규칙은 memory가 아니라 저장소 자체에 속한다.
- **다른 기기의 팀원이나 미래 세션에 도움이 될 모든 것** — 정의상 memory는 이걸 전달할 수 없다.
- **반복된 교정** — 같은 교정이 두 번 필요했다면, 그건 rule 파일을 받을 자격이 있다. memory로 쌓이게 두지 마라.

## 그래도 memory가 적절한 때

- **엄격히 개인적이고 공유할 수 없는 맥락** — 사용자의 역할, 근무 시간, 또는 어떤 공유 파일에도 넣고 싶지 않은 선호.
- **만료 시점이 분명한 일시적 cross-conversation 상태** — "사용자가 이번 주 X 마이그레이션 중" 같은 것, 날짜를 명시해서.
- **외부 시스템 참조** — 공유 rule 파일에 넣기엔 프로젝트에 너무 묶여 있지만, 매 대화마다 다시 말하기엔 너무 지속적인 것.

## 판단 순서

memory 항목을 저장하고 싶어질 때:

1. 이게 `rules/`의 rule이 될 수 있나? → rule을 대신 써라.
2. 이게 프로젝트 `CLAUDE.md`나 repo-local 문서 항목이 될 수 있나? → 그걸 대신 써라.
3. 엄격히 개인적이고 공유 불가인가? → memory가 적절하다.
4. 그 외엔 기본적으로 저장하지 마라. 다음 세션에 맥락을 다시 말하는 게, 기기들이 조용히 갈라지게 두는 것보다 싸다.

## 왜

memory의 실패 모드는 조용하다. 회사 노트북에서 잡은 가드레일이 집 노트북에는 그냥 존재하지 않는다. 사용자는 무엇이 빠졌는지 볼 수 없으니, 빠진 rule을 다시 불러낼 수도 없다. 버전 관리되는 rule 파일은 이 비대칭을 없앤다 — 모든 기기가 같은 source of truth를 보고, 개선은 `git pull`로 전파된다. memory는 협업 지식의 기본 표면이 아니라 예외여야 한다.

## 규칙

- 재사용 가능하거나 관습적이거나 공유 가능한 내용이면 memory보다 견고한 rule과 프로젝트 문서를 우선하라.
- 같은 교정을 두 번 저장하지 마라 — 재발하면 rule 파일에 속한다.
- 기존 memory 항목을 rule로 옮길 때는, 두 표면이 갈라지지 않도록 옮긴 뒤 memory를 삭제하라.
- memory가 적절한지 헷갈리면, memory로 기본 처리하지 말고 그 내용이 어디에 살아야 할지 사용자에게 물어라.
